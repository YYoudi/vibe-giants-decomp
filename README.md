## What this is

An experimental, **AI-generated** C++ reconstruction of the engine inside the
**original retail** *Giants: Citizen Kabuto* (**vanilla 1.0**, `Giants.exe`,
2000, in-house **"Amityville"** engine — x86 PE32, ~2.69 MB). The canonical
renderer is the original **DirectX 7** plugin (`gg_dx7r.dll`).

This is **not** a playable source port and **not** a drop-in replacement. It is
a big pile of C++ stubs and partially-decompiled functions, produced by an
automated loop:

```
Ghidra (decompile)  →  re-agent (LLM cleans + names + types)  →  C++ stub  →  build  →  repeat
```

> **A note on the two source trees.** The work *started* against the community
> **1.5 patch** (`GiantsMain.exe`, DX9/DX11/DX12 renderers, by `ncblakely`) —
> that produced the large `src/` pile (~6,260 functions). In June 2026 the
> canonical target **pivoted to vanilla 1.0** (the original game, DX7). `src/`
> is now **legacy reference** being rebased onto vanilla; `src_vanilla/` is the
> fresh vanilla-native reconstruction; `version_bridge/` holds the 1.5↔vanilla
> function map that drives the rebase. **Nothing here is correct just because it
> exists.**

### Honest status numbers

These are *counts*, not a quality statement.

| Metric | Value |
|---|---|
| Vanilla functions (canonical target) | ~2,574 |
| `src_vanilla/` — fresh vanilla-native track | just started (8 files) |
| `src/` — legacy 1.5-era bulk, being rebased | ~6,260 fn / ~695 files / ~97,000 lines |

The 1.5-era pipeline also recorded PASS/FAIL verdicts per function
(`RE-TRACKER.json`) — **take those verdicts with a grain of salt the size of a
house.** "pass" just means the LLM thought the decompilation round-tripped; it
is not "it is right." "fail (usable)" means even the LLM admitted it was sketchy
but the code was still dumped.

Tooling in the loop:

- **Ghidra 12.1.2** for decompilation + cross-references.
- **`re-agent`** (cloned upstream from https://github.com/Dryxio/auto-re-agent, not included) — an LLM orchestrator that
  drives Ghidra, asks the model to re-decompile/review each function, and
  produces a PASS/FAIL verdict with a parity check.
- **`ghidra-ai-bridge`** (cloned upstream, not included) — the bridge the agent
  uses to talk to a headless Ghidra.
- **MinGW-w64 (32-bit)** as the build sanity check — if the generated C++ won't
  compile as `-m32`, the loop tries to fix it.

### Architecture — the vanilla DX7 contract

The reverse target is the **original DX7 renderer**. The engine↔renderer
contract (captured dynamically + documented in `RE_docs/`):

- **Renderer plugin:** `gg_dx7r.dll`, exports `GDVSysCreate` +
  `GFXGetCapabilities` (**2 exports** — the 1.5 patch added a third, `UpCallsLoad`).
- **Engine → renderer callbacks:** **21** (texture loader, sin/cos lookup,
  localized-string lookup, time accessor, ...). Vanilla passes a **NULL
  engine-context** — there is **no COM registry** (the 1.5/DX9 path added one).
- **Per frame:** `GDVSysCreate` returns a renderer object (~55 methods); the
  engine drives a `BeginScene → Clear → scene → EndScene → Present` cycle. The
  renderer methods are `__cdecl(this, args)`, not `__thiscall`.

The 1.5-era structural call graph (game loop → render pipeline → present) is
preserved in the legacy `src/` and `CLAUDE.md` as a rough guide — its addresses
are 1.5, not vanilla. The DX9/DX11/DX12 renderers (`gg_dx9r/dx11r/dx12r.dll`)
are **patch-era** and are being reimplemented from scratch rather than
inherited; the plan is vanilla DX7 → a fresh DX12 renderer (skipping DX8/9/11).

## Building it

> Building proves the C++ parses and links. It does **not** prove the code is
> correct. Don't read a clean build as validation.

You need a **32-bit** toolchain — the original binary is PE32 x86.

**MinGW-w64 (what the pipeline uses):**
```bash
mkdir build && cd build
cmake .. -G "MinGW Makefiles" -DCMAKE_BUILD_TYPE=Debug
mingw32-make
```

**MSVC** — from an *x86* Native Tools prompt:
```bat
mkdir build && cd build
cmake .. -A Win32
msbuild GiantsRE.sln /p:Configuration=Debug
```

`CMakeLists.txt` globs `src/**/*.{cpp,h}` recursively, so new files are picked
up automatically. The `stubs/` headers provide minimal Windows/DirectX
declarations so the reversed code compiles without the DirectX SDK.

## Reproducing the setup

If you (somehow) want to reproduce the loop that produced this, you'll need to
bring the parts that aren't redistributable:

1. A legally-owned **vanilla 1.0 retail** copy of *Giants: Citizen Kabuto*
   (`Giants.exe` + `gg_dx7r.dll`). The community **1.5 patch** (DX9/DX11/DX12,
   https://giantswd.org/?file=2#984) is **optional** — only useful to compare
   against the legacy `src/`.
2. **Ghidra 12.1.2**, import `Giants.exe`, point the configs at it.
3. Clone the two tools the harness depends on (not included here):
   - `auto-re-agent` — LLM orchestrator ([https://github.com/Dryxio/auto-re-agent](https://github.com/Dryxio/auto-re-agent))
   - `ghidra-ai-bridge` — headless-Ghidra bridge ([https://github.com/Dryxio/ghidra-ai-bridge](https://github.com/Dryxio/ghidra-ai-bridge))
4. Copy [`re-agent.example.yaml`](re-agent.example.yaml) → `re-agent.yaml` and
   [`ghidra-bridge.example.yaml`](ghidra-bridge.example.yaml) →
   `ghidra-bridge.yaml`, fill in paths and your own LLM API key.

The configs in this repo are `.example` templates only — the real ones are
gitignored (machine paths + a live API key).

## Known caveats (so you don't waste your time)

- **`FAIL` ≠ garbage.** The pipeline kept `FAIL`-verdict code because it was
  often ~80–90% right. So a lot of `src/` is "good enough to compile, suspect
  in detail."
- **Global dedup is fragile.** Ghidra's `DAT_XXXXXXXX` globals got re-declared
  across translation units a lot; the pipeline `grep`s before adding, but
  duplicates and mismatches almost certainly remain.
- **CRT / SEH / init code is mostly stubbed.** `_initterm`, SEH frames, etc.
  frequently decompile to garbage and were left as stubs. That's expected.
- **Decompiler artifacts.** Things like a bogus `cos` address in the lookup
  tables have been caught before; assume more remain.
- **`src/` addresses are 1.5, not vanilla.** Don't trust any `0x00…` comment in
  `src/` as a vanilla address — those are 1.5 patch addresses. Vanilla addresses
  live in `version_bridge/VERSION_MAP_v2.csv`.
- **It's a snapshot of an autonomous run.** Another agent is still iterating on
  this directory, so files move fast and quality varies day to day.
