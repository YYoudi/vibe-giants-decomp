## What this is

An experimental, **AI-generated** C++ reconstruction of the engine inside
`GiantsMain.exe` (patch **1.520.59**, the community 1.5 patch with DX9/DX11/DX12
renderers by `ncblakely`). The 2000 game ran on the in-house **"Amityville"
engine** — x86 PE32, ~2.69 MB.

This is **not** a playable source port and **not** a drop-in replacement. It is
a big pile of C++ stubs and partially-decompiled functions, produced by an
automated loop:

```
Ghidra (decompile)  →  re-agent (LLM cleans + names + types)  →  C++ stub  →  build  →  repeat
```

### Honest status numbers

These come from `RE-TRACKER.json` and `find src/`. They are *counts*, not a
quality statement.

| Metric | Value |
|---|---|
| Functions tracked by the pipeline | ~6,260 |
| Pipeline verdict `pass` | ~6,120 |
| Pipeline verdict `fail (usable)` | ~110 |
| C++ source files in `src/` | ~695 |
| Lines of C++ in `src/` | ~97,000 |

"pass" means the LLM harness thought the decompilation round-tripped cleanly.
**Take that verdict with a grain of salt the size of a house.** "fail (usable)"
means even the LLM admitted it was sketchy but the code was still dumped.

---

Tooling in the loop:

- **Ghidra 12.1.2** for decompilation + cross-references.
- **`re-agent`** (cloned upstream from https://github.com/Dryxio/auto-re-agent, not included) — an LLM orchestrator that
  drives Ghidra, asks the model to re-decompile/review each function, and
  produces a PASS/FAIL verdict with a parity check.
- **`ghidra-ai-bridge`** (cloned upstream, not included) — the bridge the agent
  uses to talk to a headless Ghidra.
- **MinGW-w64 (32-bit)** as the build sanity check — if the generated C++ won't
  compile as `-m32`, the loop tries to fix it.

> **Why is so much not in this repo?** The two tools above, the Ghidra project,
> the exported JSON, and all game files are gitignored (too big, or not mine to
> redistribute). See [.gitignore](.gitignore) and
> [Reproducing the setup](#reproducing-the-setup) below.

### Architecture, as the AI understood it

This map reflects what the pipeline *thinks* the call
graph looks like. Trust it only as a rough guide:

```
entry (0x00643f2e)
  └─ __scrt_common_main_seh (0x00643daf)
       └─ EngineInitializeImpl (0x0062cd40)
            └─ MainGameLoop (0x0062d100)            ← the main frame loop
                 ├─ ProcessRawKeyboardInput
                 ├─ UpdateTime / ProcessMouseInput
                 ├─ ProcessGameLogic               ← render + game-state tick
                 │    ├─ BeginScene / ShutdownSubsystems (ref-counted)
                 │    ├─ EndSceneDirectional       (6-pass lighting)
                 │    ├─ SceneBegin                (sort + cull + dispatch)
                 │    ├─ DispatchGameStateTick
                 │    └─ FrameEnd                  (FPS limit + present)
                 └─ InitializeEngine              (registry + renderer + audio)
```

The renderers (`gg_dx9r.dll`, `gg_dx11r.dll`, `gg_dx12r.dll`) are plugin DLLs
exposing a common interface: `GDVSysCreate`, `GFXGetCapabilities`, `UpCallsLoad`.

## Building it

> Building proves the C++ parses and links. It does **not** prove the code is
> correct. Don't read a clean build as validation.

You need a **32-bit** toolchain — the original binary is PE32 x86.

**MinGW-w64 (what the pipeline uses):**
```bash
mkdir build && cd build
cmake .. -G "MinGW Makefiles" -DCMAKE_BUILD_TYPE=Debug
mingw32-make
```

**MSVC** — from an *x86* Native Tools prompt:
```bat
mkdir build && cd build
cmake .. -A Win32
msbuild GiantsRE.sln /p:Configuration=Debug
```

`CMakeLists.txt` globs `src/**/*.{cpp,h}` recursively, so new files are picked
up automatically. The `stubs/` headers provide minimal Windows/DirectX
declarations so the reversed code compiles without the DirectX SDK.

## Reproducing the setup

If you (somehow) want to reproduce the loop that produced this, you'll need to
bring the parts that aren't redistributable:

1. A legally-owned copy of *Giants: Citizen Kabuto* + the community 1.5 patch available here https://giantswd.org/?file=2#984.
   Put `GiantsMain.exe` somewhere the bridge can find it.
2. **Ghidra 12.1.2**, import the exe, point the configs at it.
3. Clone the two tools the harness depends on (not included here):
   - `auto-re-agent` — LLM orchestrator ([https://github.com/Dryxio/auto-re-agent](https://github.com/Dryxio/auto-re-agent))
   - `ghidra-ai-bridge` — headless-Ghidra bridge ([https://github.com/Dryxio/ghidra-ai-bridge](https://github.com/Dryxio/ghidra-ai-bridge))
4. Copy [`re-agent.example.yaml`](re-agent.example.yaml) → `re-agent.yaml` and
   [`ghidra-bridge.example.yaml`](ghidra-bridge.example.yaml) →
   `ghidra-bridge.yaml`, fill in paths and your own LLM API key.

The configs in this repo are `.example` templates only — the real ones are
gitignored (machine paths + a live API key).

## Known caveats (so you don't waste your time)

- **`FAIL` ≠ garbage.** The pipeline kept `FAIL`-verdict code because it was
  often ~80–90% right. So a lot of `src/` is "good enough to compile, suspect
  in detail."
- **Global dedup is fragile.** Ghidra's `DAT_XXXXXXXX` globals got re-declared
  across translation units a lot; the pipeline `grep`s before adding, but
  duplicates and mismatches almost certainly remain.
- **CRT / SEH / init code is mostly stubbed.** `_initterm`, SEH frames, etc.
  frequently decompile to garbage and were left as stubs. That's expected.
- **Decompiler artifacts.** Things like a bogus `cos` address in the lookup
  tables have been caught before; assume more remain.
- **It's a snapshot of an autonomous run.** Another agent is still iterating on
  this directory, so files move fast and quality varies day to day.
