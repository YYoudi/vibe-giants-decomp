# VERSION MAP v2: 1.520.59 -> vanilla 1.0 (fingerprint + multi-signal, collision-free)

Replaces `VERSION_MAP_full.csv` (the call-graph-propagation map) as the primary
hand-off artifact. Generated by `match_fingerprint.py` from `recomp_functions.jsonl`
(6254 1.5 funcs) + `vanilla_functions.jsonl` (2574 vanilla funcs), both with
relocation-invariant mnemonic fingerprints exported by `ExportFunctions.java`.

## Why v2

The previous map (461 pairs) had **20 vanilla addresses each claimed by 2-7 distinct
1.5 functions** (~11% of the map corrupted): module-shared strings (assert/error
macros) collapsed the per-function vote onto one representative, and no mutual-
uniqueness guard was applied to seeds. v2 fixes this — **1.5->many is 0 by
construction**, and the remaining many-to-1 cases are genuine merges (see below).

## Key finding: mnemonic fingerprints are WEAK cross-version

The compiler changed between vanilla retail (~MSVC 2000) and the 1.5 community patch
(~MSVC 2010s). Even the correct match has low 3-gram Jaccard:
- InitializeEngine: 1.5 size 2154 <-> vanilla size 2339, **Jaccard 0.19, prefix 2**.

So fingerprints only confirm **byte-identical** (same-compiler) functions (the 4
`fingerprint`-origin pairs, Jaccard 1.0). The robust cross-version signals are, in
order: **shared rare strings > size in bytes > call-graph topology > fingerprint**.

## PS2 triangulation (independent cross-check)

`triangulate.py` routes a third signal through the PS2 symbol dictionary: if a 1.5
func AND a vanilla func both independently string-anchor to the SAME PS2 function
(PS2 = FULL debug symbols = authoritative), they match. This is independent of the
direct 1.5<->vanilla comparison.

- vanilla->PS2 string matches are sparse (54 candidates, 14 conf>=0.4) — vanilla has
  few string refs and PS2 has only ~223 funcs with string refs — so call-graph
  propagation gets no traction (0 grown). This is the honest ceiling.
- Result: **8 v2 pairs independently CONFIRMED** (+`triangulated` flag), **1 NEW**
  reliable pair (AttackObject), **1 conflict flagged** (chat_init: v2's vanilla
  00538820 is correct via 6 ChatColor strings; the PS2 leg matched a different vanilla
  func on TextColor strings — v2 wins, conflict kept for the record).

## Transitive PS2 naming — the main payoff

For every reliable v2 pair whose 1.5 side has a PS2 name (from the 1.5->PS2 match
set), the **vanilla address inherits that authoritative name**. **100 reliable pairs
(39%) carry a PS2 name** in the `ps2_name` column — e.g. vanilla `0050a720`=
`teammecc_set_mode`, `00538820`=`chat_init`, `004d31d0`=`reaper_check_enter_jet_ski`,
`004fdf20`=`smartie_get_name`. The main agent reversing vanilla gets real names/types
for 100 functions out of the box.

## How to use (main agent)

- For EXACT/HIGH/MEDIUM rows in `VERSION_MAP_v2.csv`: cite **vanilla_addr** as canonical
  provenance; carry the 1.5 reverse (semantics, types, PS2 name) onto it. Verify MEDIUM
  bodies against vanilla. EXACT/HIGH transfer directly.
- For LOW rows: treat the vanilla_addr as a **starting hint**; confirm in Ghidra before
  citing. (These matched on size + caller-count only — ~half will be right.)
- For merge rows: the single vanilla function contains several 1.5 functions' worth of
  logic. Reverse the vanilla function once, then split per the 1.5 members.
- `ps2_name` column gives the PS2 symbol (when the 1.5 func was matched to PS2) for
  naming/types.
